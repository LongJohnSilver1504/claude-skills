# Pipeline Help

## Overview

Interactive guide for the feature development pipeline. Explains each stage, what it produces, and how to navigate the flow.

> **Maintenance:** Regenerate this file whenever any pipeline skill's routing changes.

## The Pipeline

```
brainstorm → generate-prd → prd-clarifier → prd-to-ux → plan-implementation → execute-tasks → finish-feature → generate-feature-doc
     │                                           │               │
     └──────────── prototype (optional) ────────┘               └── to-issues (optional, instead of execute-tasks)
```

**Optional branches:**
- `/prototype` — flesh out a design before committing to it (throwaway terminal app for state/logic questions, or several UI variations on one route). Branch off during brainstorm or UX design, then return to the pipeline.
- `/to-issues` — convert the implementation plan into tracker issues instead of executing locally. Replaces `execute-tasks` when the work will be picked up asynchronously.

## Pipeline Stages

| # | Skill | Input | Output | Can Skip? |
|---|-------|-------|--------|-----------|
| 1 | `/brainstorm` | Rough idea | Validated idea + approach (`DESIGN.md`) | Yes — if the feature is already clear |
| 2 | `/generate-prd` | Feature idea | `PRD.md` | No — entry point for specs |
| 3 | `/prd-clarifier` | PRD | Clarified PRD (clarifications merged into `PRD.md`) | Yes — if PRD is clear |
| 4 | `/prd-to-ux` | PRD | `UX-spec.md` (9 passes inc. test matrix) | No |
| 5 | `/plan-implementation` | All specs | Implementation plan | No |
| 6 | `/execute-tasks` | Plan | Working code (subagents + review gates) | Yes — use `/to-issues` to farm out instead |
| 7 | `/finish-feature` | Code + tests | Tests/build run, commit / PR / discard | No |
| 8 | `/generate-feature-doc` | Feature code | `README.md` + artifact cleanup | Yes — for small features |

## Quick Start

**Full feature (recommended):**
```
/brainstorm (or /generate-prd if the idea is clear) → follow the skill transitions
```

**Small change (skip early stages):**
```
/plan-implementation → /execute-tasks → /finish-feature
```

**Bug fix:**
```
Just fix the bug. No pipeline needed.
```

## Artifacts

Artifacts live co-located with the feature.

### What stays (permanent)
- `PRD.md` — in feature folder (includes embedded Clarifications)
- `UX-spec.md` — in feature folder
- `README.md` — in feature folder (generated by `/generate-feature-doc`)

### What gets deleted (intermediate)
Cleaned up after the pipeline completes:
- `PRD-*.md`
- `*-implementation-plan.md`
- `*-test-plan.md`

## Resuming After Context Cleanup

When you clean context mid-pipeline:

1. **Tell Claude which feature you're working on** — e.g., "I'm working on reservation-details"
2. **Invoke the skill you need** — Claude will read the pipeline state automatically
3. **Each skill checks for existing artifacts and restores context:**
   - `DESIGN.md` (brainstorm output)
   - `PRD.md` with embedded Clarifications
   - `UX-spec.md`
   - The implementation plan
   - `PROGRESS.md` (execution state)

You don't need a special command. Just invoke the next skill and it handles resume.

## Common Questions

**Q: Do I need to run every stage?**
A: No. The pipeline is a guide, not a mandate. Invoke whichever skill fits your current need.

**Q: What if I want to add requirements mid-pipeline?**
A: Just tell Claude. Update the PRD and downstream artifacts as needed.

**Q: How do I build visually polished components or improve existing ones?**
A: `/refactoring-ui` auto-triggers when you ask to build or review UI. It applies Refactoring UI principles (hierarchy, typography, spacing, color, depth, polish) when writing JSX, and audits existing components with prioritized findings (🔴 Critical / 🟡 Important / 🟢 Nitpick) and rule citations.
